# Cloud-base / wet-layer research harness

Reproducible evidence behind [`docs/cloud-base-research.md`](../../docs/cloud-base-research.md).
**Not production code** — nothing here is imported by the app. The analysis reuses the real
app domain code (`src/domain/*`, `src/data/openMeteo`) against committed fixtures.

## Run

```bash
# 1) (optional) refresh fixtures from live METAR + Open-Meteo  — needs network
npm run research:cloud-base:fetch

# 2) run the comparison (skipped in the normal suite; writes RESULTS.txt)
npm run research:cloud-base
```

The analysis is deterministic against `fixtures/*.json`, so step 2 reproduces `RESULTS.txt`
without network. Step 1 overwrites the fixtures with a fresh live snapshot.

## Files

| File | Purpose |
|------|---------|
| `fetch.mjs` | Snapshot METAR + Open-Meteo profiles for a diverse station list → `fixtures/`. |
| `methods.ts` | Pure candidate estimators: Espy (A), env-saturation (B), profile-aware LCL (C), model cloud (D). |
| `cloudBase.research.test.ts` | Harness: reuses app domain code, runs A–E vs METAR, prints tables. |
| `RESULTS.txt` | Captured output (regenerated by step 2). |
| `fixtures/` | Raw upstream JSON per station + `manifest.json`. |

## Headline finding

For the shallow **0–500 m** band that actually wets a drone (fog, marine/radiation stratus,
drizzle layers), the **surface dew-point signal beats the coarse model profile** — the model
smooths low layers away. The model profile helps for *elevated* moist layers and broader
context, not for pinpointing a low base. See the research note for the full argument and the
implementation proposal.
